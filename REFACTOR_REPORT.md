# Baby Steps Prototype Refactor Report

Date: 2026-06-18

## Audit Summary

Baby Steps is an Expo React Native prototype using `expo-router`, TypeScript, NativeWind/Tailwind, Supabase, AsyncStorage, Expo AV, and Jest via `jest-expo`. The package manager is npm, with `package-lock.json` present. The main entry point is `expo-router/entry`; the runtime root is `app/_layout.tsx`.

The prototype already contains the main product surfaces expected for an MVP foundation: onboarding, auth, parent dashboard, child profiles, child mode, stories, games, museum screens, progress/activity logging, and achievements. The main scalability risks are hardcoded content, very large game/story components, duplicated story logic, prototype routing shortcuts, broad local state, and limited test coverage.

Important findings:

- App routing is file based under `app/`, with parent, child, story, game, museum, auth, and onboarding routes.
- Supabase currently covers auth-backed children, activities, achievements, and child achievements through `schema.sql`.
- Games, stories, lessons, translations, museum copy, onboarding copy, media references, and quizzes are still mostly hardcoded.
- Several routes referenced from settings are not implemented yet: `/content-management`, `/privacy-settings`, `/help-support`, and `/account-info`.
- `components/SkipButtonOnboarding.tsx` appears unused and points to stale `/add-child` routing.
- `expo-av` works today but now emits a deprecation warning during export and should be planned for replacement with `expo-audio` / `expo-video`.
- Lint still reports warnings for unused imports/variables, duplicate imports, hook dependency arrays, and unreachable code.

## Current Architecture Overview

- Framework: Expo SDK 54, React 19, React Native 0.81, Expo Router 6.
- Package manager: npm.
- Routing: `expo-router` file routes in `app/`.
- Styling: NativeWind/Tailwind plus inline React Native styles.
- Auth/backend: Supabase client in `lib/supabase.ts`.
- Local persistence: AsyncStorage, mostly in game progress managers and utility files.
- Testing: Jest with `jest-expo`.
- Database context: `schema.sql` with `children`, `activities`, `achievements`, and `child_achievements`.
- Main app root: `app/_layout.tsx`, which currently handles fonts, splash state, onboarding state, Supabase session, background audio, app state, and root navigation redirects.

## Known Risks

- Lint warnings remain and should be cleaned incrementally.
- No full UI, navigation, device, or E2E tests exist yet.
- Story components are duplicated and large, increasing the risk of inconsistent fixes.
- Game components are very large and mix UI, content, scoring, audio, achievements, persistence, and animations.
- Settings links to future screens that do not exist yet.
- Some text/media appears to have encoding issues from the prototype era.
- Translation lookup is exact-string based, so dynamic text and slightly changed strings do not translate reliably.
- Supabase schema is context-only and explicitly warns it may not be executable as-is.

## Remaining Technical Debt

- Split root layout responsibilities into smaller hooks/services.
- Consolidate story screens around a shared data-driven story player.
- Split game components into content, game state, scoring, rendering, audio, and persistence modules.
- Replace `any` in storage, museum content, icon names, and image/audio maps.
- Normalize progress and activity models before adding payments or premium access.
- Add integration tests for parent/child navigation and game completion flows.
- Add route guards for parent/child access and missing/future settings screens.
- Replace `expo-av` before SDK upgrades make it blocking.

## Payment Feasibility

Sources checked on 2026-06-18:

- Apple App Store Review Guidelines: https://developer.apple.com/app-store/review/guidelines/
- Apple auto-renewable subscriptions: https://developer.apple.com/app-store/subscriptions/
- Google Play Billing: https://developer.android.com/google/play/billing
- Google Play subscriptions: https://developer.android.com/google/play/billing/subscriptions
- Google Play Billing Library release notes: https://developer.android.com/google/play/billing/release-notes
- Google Play Billing Library version deprecation FAQ: https://developer.android.com/google/play/billing/deprecation-faq

Features that may eventually need payments:

- Premium lessons, stories, games, and curriculum packs.
- Subscription access to a full content library.
- One-time unlocks for story/game packs.
- Parent/teacher dashboard features.
- School or institution plans.
- Content creator/admin publishing tools, if monetized.

Likely model:

- For consumer mobile app digital content, plan around in-app purchases and subscriptions.
- A subscription is likely the cleanest model if Baby Steps will keep adding lessons, stories, games, and curriculum updates.
- One-time unlocks can work for fixed packs, but they create more entitlement complexity.
- School/institution plans may be sold outside the app if they are organization-level services, but consumer/family purchases inside the app likely need store-compliant IAP.

Android requirements:

- Use Google Play Billing for digital products/subscriptions sold in the Android app.
- Configure products/subscriptions in Play Console.
- Integrate a React Native/Expo-compatible billing library.
- Add backend verification through Google Play Developer API.
- Track purchase tokens, order IDs, subscription state, renewals, cancellations, and entitlement status.
- Note: Google Play Billing Library version requirements change. As of the 2026-06-18 verification pass, Google lists Play Billing Library 9.0.0 as available, and the deprecation FAQ lists the new app/update deadline for version 7 as 2026-08-31, version 8 as 2027-08-31, and version 9 as 2028-08-31.

iOS requirements:

- Use Apple In-App Purchase / StoreKit for digital features, premium content, game levels, subscriptions, and full unlocks sold in-app.
- Configure IAP products/subscriptions in App Store Connect.
- Use StoreKit 2 or a compatible React Native/Expo bridge.
- Add backend verification through App Store Server API and App Store Server Notifications.
- Support restore purchases and subscription status display.
- Clearly describe subscription terms before purchase.

Backend verification needed:

- Receipt / transaction validation for Apple.
- Purchase token validation for Google.
- Webhook/notification handling for renewals, refunds, chargebacks, grace periods, and cancellations.
- Server-owned entitlement calculation instead of trusting the client.
- Idempotent transaction processing.
- Audit logs for entitlement changes.

Likely payment database entities:

- `users`
- `plans`
- `products`
- `prices`
- `purchases`
- `subscriptions`
- `transactions`
- `entitlements`
- `entitlement_grants`
- `platform_receipts`
- `access_control_rules`
- `organizations` / `schools`
- `organization_memberships`

Architectural changes before payments:

- Create a server/API layer. Do not implement payment verification only on-device.
- Normalize user and child identity.
- Add an entitlement service used by content selection and route guards.
- Define content IDs before monetization, so purchases unlock stable IDs rather than component names.
- Add analytics/events for paywall views, purchase starts, purchase results, and entitlement changes.
- Add tests for entitlement calculation and access control.

## Database Content Feasibility

Current hardcoded content locations:

- Games:
  - `content/games/countingGameStages.ts`
  - `content/games/lugandawords.ts`
  - `content/games/wordgamewords.ts`
  - game component state/config in `components/games/*.tsx`
- Stories:
  - `components/stories/*Story.tsx`
  - routes under `app/child/stories/*.tsx`
- Lessons:
  - `content/games/lugandawords.ts`
- Questions and answers:
  - story quiz data inside individual story components.
  - game prompts/options inside hardcoded game content and game components.
- Curriculum structure:
  - implicit in tabs, game cards, stages, levels, and story lists.
- Images/audio/media:
  - bundled under `assets/`, referenced by `require()` and string names.
- Translations:
  - `lib/translations.ts`

Current schema support:

- Supports children, activities, achievements, and child achievement joins.
- Supports basic activity history with type, score, duration, details, stage, and level.
- Does not yet model curriculum content, story pages, lesson levels, game definitions, media assets, localization, content versioning, publication status, creators, review workflow, or entitlements.

Schema gaps:

- `activities.details` is text; future activity metadata should likely be `jsonb`.
- `activities.score` is text; future scoring may need numeric score, max score, percent, and outcome fields.
- There are no content IDs to link activities/progress to lessons, stories, games, or curriculum nodes.
- There is no parent profile table separate from `auth.users`.
- There are no roles or organization/school tables.
- There is no media asset table for image/audio URLs, alt text, attribution, and CDN metadata.
- There is no localization table for English/Luganda and future languages.
- There is no content workflow for drafts, review, publish, archive, version, or creator attribution.

Suggested future schema additions:

- `curricula`, `curriculum_units`, `lessons`, `lesson_items`
- `stories`, `story_pages`, `story_questions`, `story_options`
- `games`, `game_levels`, `game_prompts`, `game_options`
- `media_assets`, `content_media`
- `localizations`
- `progress_records`, `completion_records`
- `content_versions`, `content_reviews`
- `creator_profiles`, `content_assignments`
- payment/entitlement tables listed above

Migration difficulty:

- Medium for Luganda lesson/word content because it is already partly structured.
- Medium for counting and word game content after the new `content/games/` boundary.
- High for stories because page, quiz, audio, highlighting, and UI logic are repeated inside components.
- High for games if gameplay rules remain embedded in large components instead of consuming typed content contracts.

Recommended path:

- Define TypeScript content contracts first.
- Add API/database DTOs matching those contracts.
- Migrate one content type at a time, starting with Luganda lessons or word game levels.
- Keep bundled fallback content while API content is introduced.
- Add content versioning and offline cache before removing hardcoded fallbacks.

## 8-Week MVP Preparation Roadmap

Week 1: Research, audit, cleanup, and validation

- Finish lint warning cleanup in high-risk files.
- Confirm target audience, parent/teacher needs, curriculum goals, and cultural content priorities.
- Validate existing content quality and language accuracy.

Week 2: Refine needs and MVP scope

- Apply MoSCoW/Kano prioritization.
- Decide which games/stories/lessons ship in MVP.
- Define must-have progress tracking and parent dashboard metrics.
- Define content contracts for stories, games, lessons, and media.

Week 3: Core implementation and prototype revamp

- Split large game/story components safely.
- Add route guards and missing route placeholders.
- Stabilize auth, child profile selection, and parent/child switching.
- Add focused UI smoke tests.

Week 4: Database-driven content foundations

- Add content schema migrations and API access patterns.
- Migrate one low-risk content type first.
- Add content fallback/cache behavior.
- Start admin/creator workflow design, not full implementation.

Week 5: Game/story/lesson polish and progress tracking

- Normalize scoring and completion events.
- Tie progress to content IDs.
- Add achievement tests and progress tests.
- Improve audio/media handling and accessibility.

Week 6: Piloting, feedback, bug fixes, and iteration

- Pilot with parents/children/teachers.
- Collect usability, comprehension, and content feedback.
- Fix navigation, onboarding, game difficulty, and story pacing issues.

Week 7: Marketing, onboarding, analytics, and deployment preparation

- Add analytics events with privacy review.
- Improve onboarding and parent guidance.
- Prepare app store assets, privacy disclosures, and data deletion flows.
- Decide payment launch timing, but avoid rushed payment implementation.

Week 8: Final QA, launch readiness, deployment, and maintenance planning

- Run device QA on Android and iOS.
- Run regression test pass and fix blocking bugs.
- Prepare deployment checklist and rollback plan.
- Finalize post-launch support, monitoring, and content update process.
